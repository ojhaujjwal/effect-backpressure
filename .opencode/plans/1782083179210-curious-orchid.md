# Plan: Shared `ojhaujjwal/oxlint-rules` Package

## Architecture

Following the `@effect/oxc` pattern from `~/.local/share/effect-solutions/effect-smol/packages/tools/oxc/`:

- One standalone npm package that exports an oxlint plugin + optional shared base config
- Both `tesla-charger` and `media-diy-backend` consume it as `"jsPlugins": ["ojhaujjwal/oxlint-rules/oxlint"]`
- No bundler — `tsc` alone is sufficient. Oxlint plugins are plain Node.js modules with a default-export object of rules; oxlint `import()`s them natively. The `tsc` build produces pure ESM (no CJS).

## Step-by-Step Implementation

### 1. Initialize Repo

`pnpm init`. Set `"type": "module"`.

### 2. `package.json`

```jsonc
{
  "name": "ojhaujjwal/oxlint-rules",
  "version": "0.1.0",
  "type": "module",
  "exports": {
    "./package.json": "./package.json",
    "./oxlint": "./src/oxlint/index.ts",
    "./oxlint/rules/*": "./src/oxlint/rules/*.ts",
    "./oxlintrc.json": "./oxlintrc.json"
  },
  "publishConfig": {
    "exports": {
      "./package.json": "./package.json",
      "./oxlint": "./dist/oxlint/index.js",
      "./oxlint/rules/*": "./dist/oxlint/rules/*.js",
      "./oxlintrc.json": "./oxlintrc.json"
    }
  },
  "files": ["oxlintrc.json", "src/**/*.ts", "dist/**/*.js", "dist/**/*.d.ts"],
  "scripts": {
    "build": "tsc -b tsconfig.json",
    "test": "vitest"
  },
  "dependencies": {
    "@oxlint/plugins": "^1.68.0"
  },
  "peerDependencies": {
    "typescript": ">=5.0.0"
  },
  "devDependencies": {
    "@types/node": "^22",
    "oxlint": "^1.68.0",
    "typescript": "^5.8",
    "vitest": "^3"
  }
}
```

**Dependency rationale:**
- `@oxlint/plugins` (runtime dep): provides `defineRule` / `definePlugin` used by all ported rules. Alternative: drop it and use plain-object exports like `@effect/oxc` does — but that means rewriting all 23 rules. Keep it to minimize porting friction.
- `oxlint` (devDep): provides `CreateRule`, `Visitor`, `ESTree`, `Fixer` types used in test utils and rule type annotations.
- `peerDependencies.typescript`: rules export raw `.ts` source in dev mode. During dev consumption, oxlint needs TypeScript to transpile on-the-fly (consuming projects use `NODE_OPTIONS='--import tsx/esm'`).

### 3. `tsconfig.json`

Identical to `@effect/oxc`'s effective config (no `jsx` or `@effect/language-service`):

```jsonc
{
  "$schema": "http://json.schemastore.org/tsconfig",
  "include": ["src"],
  "compilerOptions": {
    "outDir": "${configDir}/dist",
    "rootDir": "${configDir}/src",
    "incremental": true,
    "composite": true,
    "target": "ES2022",
    "module": "NodeNext",
    "moduleDetection": "force",
    "verbatimModuleSyntax": true,
    "rewriteRelativeImportExtensions": true,
    "erasableSyntaxOnly": true,
    "declarationMap": true,
    "sourceMap": true,
    "strict": true,
    "exactOptionalPropertyTypes": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true,
    "skipLibCheck": true,
    "noErrorTruncation": true,
    "types": ["node"]
  }
}
```

**Import extensions:** Use `.ts` extensions with `rewriteRelativeImportExtensions: true` (the Effect convention). Source files import with `.ts` — tsc rewrites to `.js` in `dist/`. The existing rules in tesla-charger and media-diy use `.js` imports — those must be changed to `.ts` when porting.

### 4. `vitest.config.ts`

```ts
import { defineConfig } from "vitest/config"

export default defineConfig({
  test: {
    include: ["test/**/*.test.ts"]
  }
})
```

No workspace-shared config to extend (unlike `@effect/oxc` which extends `../../../vitest.shared.ts`). Standalone is simpler.

### 7. Test Infrastructure

Use `RuleTester` from `oxlint/plugins-dev` (ships in `oxlint` ≥1.70). It parses real source code, runs the rule, and verifies diagnostics — no hand-crafted AST stubs.

```ts
// test/no-process.test.ts
import { RuleTester } from "oxlint/plugins-dev"
import rule from "../src/oxlint/rules/no-process.ts"

const tester = new RuleTester()

tester.run("no-process", rule, {
  valid: [
    // process.argv is allowed
    "process.argv",
    // using Config instead of process.env
    `import { Config } from "effect"; Config.string("FOO")`,
    // accessing other globals is fine
    "console.log('hello')"
  ],
  invalid: [
    {
      code: "process.env.FOO",
      errors: [{ message: /process/ }]
    },
    {
      code: "globalThis.process.env.FOO",
      errors: [{ message: /process/ }]
    }
  ]
})
```

**No test utils file needed** — `RuleTester` provides the full ESLint-compatible test API itself.

For rules that accept options:
```ts
tester.run("rule-with-options", rule, {
  valid: [{ code: "...", options: [{ allowFoo: true }] }],
  invalid: [{ code: "...", errors: [{ message: "..." }], options: [] }]
})
```

The `RuleTester` API:
- `valid`: array of `string | { code, options?, settings?, filename? }`
- `invalid`: array of `{ code, errors: Array<{ message, messageId, line?, column? }>, output? }`

### 9. Migrate Consuming Projects

In both `media-diy-backend` and `tesla-charger`:

1. Remove `lint/` directory (custom rules and `index.ts`)
2. Remove `@oxlint/plugins` from devDependencies
3. Add `"ojhaujjwal/oxlint-rules": "^0.1.0"` to devDependencies
4. Update `.oxlintrc.json`:
   - Change `"jsPlugins": ["./lint/index.ts"]` → `"jsPlugins": ["ojhaujjwal/oxlint-rules/oxlint"]`
   - Change rule prefixes: `tesla-charger/` → `effect-lint/`, `media-diy/` → `effect-lint/`
   - Keep any project-specific overrides (whitelisted files, rules set to "warn" vs "error")
5. For rules previously excluded from `media-diy` (e.g., `no-raw-date`), either enable them or keep them off
6. Run `pnpm lint` and verify zero violations
